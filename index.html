PRACTICAL-5

Aim: Demonstrate synchronous and asynchronous communication.

PULSES:

• Inter-Process Communication (IPC)
• Synchronous communication (Blocking) vs Asynchronous communication (Non-blocking)
• Part A: Create pipe using os.pipe(), fork() process, write data, Child blocks and waits.
• Part B: Set read end to non-blocking mode, Child continues even if data not available.

Program:

import os, fcntl, time

# Setup a communication pipe
read_end, write_end = os.pipe()

# --- PART A: Synchronous / Blocking Simulation ---
print("[SYNC] Child: Waiting for data (Blocking Read)...")
# Note: In a real run, os.read() pauses the program until data exists
# We simulate the delay to show the "waiting" state.
time.sleep(1)
os.write(write_end, b"Sync_Message")
data = os.read(read_end, 100).decode()
print(f"[SYNC] Child: Received -> {data}\n")

# --- PART B: Asynchronous / Non-blocking Simulation ---
# Change the file descriptor flags to Non-Blocking
flags = fcntl.fcntl(read_end, fcntl.F_GETFL)
fcntl.fcntl(read_end, fcntl.F_SETFL, flags | os.O_NONBLOCK)

print("[ASYNC] Child: Checking for data immediately (Non-blocking)...")
try:
    data = os.read(read_end, 100) # This will not wait
except BlockingIOError:
    print("[ASYNC] Result: No data available. Child continues other execution tasks.")

Output:

[SYNC] Child: Waiting for data (Blocking Read)... (1 second pause observed) [SYNC] Child: Received -> Sync_Message [ASYNC] Child: Checking for data immediately (Non-blocking)... [ASYNC] Result: No data available. Child continues other execution tasks.
1. Synchronous communication means: B) Receiver waits until message arrives
2. Which type of communication is blocking in nature? C) Synchronous
3. In asynchronous communication, the receiver: C) Continues execution without waiting
Resolution: I successfully distinguished between blocking and non-blocking calls. Synchronous communication ensures data readiness before proceeding, while asynchronous allows process concurrency.

PRACTICAL-6

Aim: Implement a simple RPC system using Python (XML-RPC).

PULSES:

• Remote Procedure Call (RPC) Basics • Python xmlrpc library
• Create RPC Server, Register functions (add, greet), and listen.
• Create RPC Client, call remote functions, and receive output transparently.

Program:

# --- server.py ---
from xmlrpc.server import SimpleXMLRPCServer

def add_numbers(a, b): return a + b
def say_hello(name): return f"Hello, {name}! This is the RPC Server."

server = SimpleXMLRPCServer(("localhost", 8000))
server.register_function(add_numbers, "add")
server.register_function(say_hello, "greet")
print("RPC Server is listening on port 8000...")
server.serve_forever()

# --- client.py ---
import xmlrpc.client
proxy = xmlrpc.client.ServerProxy("http://localhost:8000/")
print(f"RPC Add Result (10+5): {proxy.add(10, 5)}")
print(f"RPC Greet Result: {proxy.greet('Kali-User')}")

Output:

[Terminal 1] RPC Server is listening on port 8000... [Terminal 2] RPC Add Result (10+5): 15 RPC Greet Result: Hello, Kali-User! This is the RPC Server.
1. RPC stands for: B) Remote Procedure Call
2. Main advantage of RPC? B) Hides network communication details
3. XML-RPC uses which protocol? C) HTTP
4. Python class to create server? C) SimpleXMLRPCServer
Resolution: I learned to build a distributed application where the client executes code on a remote server as if it were a local function call.

PRACTICAL-7

Aim: Client remotely invokes procedures on server.

PULSES:

• Client calls a function executed on a remote server.
• Client sends a request (procedure name + parameters).
• Server executes procedure, results are sent back.
• Transparency: Client does not need to know where/how procedure is executed.

Program:

# --- rpc_server_p7.py ---
from xmlrpc.server import SimpleXMLRPCServer

# Remote function to calculate factorial
def find_factorial(n):
    import math
    return math.factorial(n)

server = SimpleXMLRPCServer(("127.0.0.1", 9000))
server.register_function(find_factorial, "remote_fact")
print("Remote Invocation Server ready...")
server.serve_forever()

# --- rpc_client_p7.py ---
import xmlrpc.client
conn = xmlrpc.client.ServerProxy("http://127.0.0.1:9000/")
number = 6
print(f"Requesting remote factorial of {number}...")
result = conn.remote_fact(number)
print(f"Result returned from remote server: {result}")

Output:

[Server Console] Remote Invocation Server ready... [Client Console] Requesting remote factorial of 6... Result returned from remote server: 720
1. Client remotely invokes procedures refers to: C) Remote Procedure Call
2. Where does procedure execute in RPC? C) Server machine
3. Which architecture is used in RPC? B) Client-Server
Resolution: Observed the transparency property of RPC, where procedural calls are marshaled over the network and executed on a separate server host.

PRACTICAL-8

Aim: Simulate Cristian's and Berkeley algorithms using Python.

PULSES:

• Clock synchronization in distributed systems.
• One machine acts as a time server.
• Client requests time and adjusts its clock considering RTT delay.

Program:

import time

def simulate_cristian_algorithm():
    # Simulation: Time server is 1.5 seconds ahead
    server_time = time.time() + 1.5 
    
    # Client sends request (Start Time)
    t_start = time.time()
    
    # Network Round Trip Delay (RTT Simulation)
    time.sleep(0.4) 
    
    # Client receives response (End Time)
    t_end = time.time()
    rtt = t_end - t_start
    
    # Sync Logic: T_client = T_server + (RTT / 2)
    synchronized_time = server_time + (rtt / 2)
    
    print(f"Measured RTT: {round(rtt, 3)} seconds")
    print(f"Adjusted Sync Time: {time.ctime(synchronized_time)}")

simulate_cristian_algorithm()

Output:

kali@kali:~$ python3 sync_sim.py Measured RTT: 0.402 seconds Adjusted Sync Time: Thu Oct 24 18:20:45 2024
1. Cristian's algorithm architecture? A) Master-Slave (or B Client-Server)
2. Client adjusts clock using? B) Round Trip Time (RTT)
3. Berkeley algorithm used in? B) Distributed systems
Resolution: I simulated clock synchronization. I learned that Cristian’s algorithm compensates for network propagation delay by adding half of the RTT to the server’s timestamp.
